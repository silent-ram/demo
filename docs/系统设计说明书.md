## 1. 系统总体设计

### 1.1 设计原则

- **微服务拆分原则：** 按业务领域划分服务边界，各服务独立部署、独立数据库；
- **单一职责原则：** 每个微服务只负责一个业务域；
- **接口隔离原则：** 服务间通过 OpenFeign 定义的接口通信，不共享数据库；
- **防御性设计：** 关键服务间调用引入 Resilience4j 断路器，避免级联故障。

### 1.2 系统整体架构

└──┬──────────┬──────────┬──────────┬────────────────────── ┘
   │          │          │          │

user-    device-    data-     alert-

   │          │       │          │

### 2.2 InfluxDB 数据模型

- **Bucket：** `device_metrics`
- **Measurement：** `sensor_data`
- **Tags：** `device_id`（string）, `device_name`（string）
- **Fields：** `temperature`（float, °C）, `vibration`（float, mm/s）, `pressure`（float, MPa）, `fault_probability`（float, 0\~1）
- **时间精度：** 毫秒（ms）
- **数据保留策略：** 30天

**写入示例（Line Protocol）：**

```
sensor_data,device_id=1,device_name=压缩机A temperature=58.3,vibration=0.45,pressure=1.3,fault_probability=0.82 1711900000000000000
```

## 4. 关键模块设计

### 4.1 JWT 认证流程

```
客户端                    gateway-service              user-service
  │                            │                           │
  │── POST /user/login ────────►│── 路由转发 ──────────────►│
  │                            │                     验证用户名密码
  │◄── {token: "eyJ..."} ──────│◄──── 返回JWT ────────────│
  │                            │                           │
  │── GET /api/device/list ───►│                           │
  │   Header: Authorization:   │ JWT全局过滤器验证token      │
  │   Bearer eyJ...            │ 解析userId/role注入header  │
  │                            │── 转发至device-service ───►│
```

### 4.2 Logistic 回归模型设计

**特征工程：**

| 特征               | 说明        | 正常范围     | 异常范围 |
| ---------------- | --------- | -------- | ---- |
| temperature      | 设备温度（°C）  | 40\~60   | >75  |
| vibration        | 振动值（mm/s） | 0.1\~0.3 | >0.8 |
| pressure         | 压力值（MPa）  | 1.0\~1.5 | >2.0 |
| temp\_vib\_ratio | 温度×振动交叉特征 | -        | -    |
| pressure\_diff   | 压力与均值偏差   | -        | -    |

**训练数据模拟策略：**

- 正常样本（label=0）：三个特征在正常范围内高斯分布，占 70%
- 故障样本（label=1）：至少一个特征超出阈值，占 30%
- 总样本量：5000 条
- 训练集/测试集：8:2 划分

**模型参数：**

- 算法：LogisticRegression（sklearn）
- 求解器：lbfgs
- 正则化：L2，C=1.0
- 最大迭代：1000
- 标准化：StandardScaler（与模型一同保存）

**模型文件：**

- `ml-service/model/logistic_model.pkl`（训练后的模型）
- `ml-service/model/scaler.pkl`（特征标准化器）

### 4.3 实时预警数据流

```
data-collector-service
    │ 每5秒生成传感器数据
    │ 写入 InfluxDB
    │ Feign调用 alert-service
    ▼
alert-service
    │ 调用 ml-service /predict
    │ 获取 fault_probability
    │ 与阈值比较（从t_config读取）
    │ 超阈值 → 写入 t_alert
    │ 更新设备状态（Feign→device-service）
    │ WebSocket推送至前端
    ▼
前端 /monitoring 页面
    └── 实时更新故障概率进度条 + 弹出告警通知
```
